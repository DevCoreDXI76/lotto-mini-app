# 로또 미니앱 MVP — 개발 스펙 (PRD)

> 이 문서는 VSCode의 Claude Code(또는 다른 AI 코딩 에이전트)에 그대로 전달해서 개발을 시작하기 위한 문서입니다. 비개발자가 AI 코딩 툴로 작업한다는 전제로, 기술 스택과 범위를 최대한 좁고 구체적으로 잡았습니다.

## 1. 문제 정의

로또 번호 생성기 앱은 이미 시장에 많지만, 대부분 "번호 생성" 기능 하나에만 집중되어 있고 실제로 도움이 되는 정보(가까운 고배출 판매점)는 잘 노출되지 않는다. 사용자는 재미로 다양한 방식의 번호를 뽑아보고 싶어 하면서, 동시에 "어디서 사야 하는지"에 대한 실용적인 궁금증도 갖고 있다. 이 둘을 하나의 가벼운 유틸리티 앱으로 묶는다.

**중요한 법적 전제**: 이 앱은 당첨 확률을 높여준다고 주장하지 않는다. 번호 생성은 재미/통계적 흥미 요소이고, 모든 화면에 "당첨을 보장하지 않습니다" 류의 디스클레이머를 노출한다. 유료 결제·환불 보장 문구는 사용하지 않는다(무료 서비스). 번호별로 우열을 암시하는 표현(별점, 순위, "추천" 등)도 사용하지 않는다 — 자세한 원칙은 F1/F5 참고.

## 2. 목표

- V1을 6주 이내(주당 약 21시간 투입 기준)에 웹앱 형태로 배포하고 실사용자 트래픽을 받는다.
- "가까운 판매점 찾기" 기능의 실사용률(방문자 대비 위치 조회 클릭 비율) 30% 이상을 1차 성공 기준으로 삼는다.
- 번호 생성기 재사용률(같은 세션에서 2회 이상 생성) 20% 이상.
- 법적 리스크 없이(과장 광고 문구, 결제 유도, 번호별 우열 암시 없이) 운영한다.

## 3. Non-goals (V1 범위 제외)

- 회원가입/로그인 — 초기엔 익명 사용, 즐겨찾기 등은 로컬 저장(localStorage)으로 대체.
- 유료 결제, 프리미엄 기능 — 법적 리스크 회피 및 속도를 위해 완전 무료로 시작.
- 카카오톡/텔레그램 미니앱 네이티브 배포 — V1은 반응형 웹앱(PWA)으로만 배포하고, 반응이 확인되면 V2에서 검토.
- 전국 판매점 100% 실시간 동기화 — 데이터 소스가 주간 업데이트라 실시간 동기화는 불가능하며 시도하지 않는다.
- 번호 자동 구매/연동 — 동행복권 구매 대행은 법적으로 별도 이슈이므로 제외. "번호 복사하기" 정도만 지원.
- LLM 기반 번호 생성 — 13절 참고. 번호 생성 로직은 규칙 기반 통계로만 구현한다.
- 참고 사이트(e-ubis.com 등)의 정확한 가중치 배율을 그대로 복제하지 않는다 — 조합 비율은 이 문서(F1)에 정의된 값을 따른다.
- 근거/특징 문구를 그럴듯하게 지어내지 않는다 — 계산 불가능한 항목은 표시하지 않는다.
- 번호별 별점·순위·점수 같은 정량적 우열 표시는 사용하지 않는다 — 이유는 F1 참고.
- F2(예산 기반)·F3(역대 통계)·F4(판매점 찾기)에는 F1의 리치 UI(전략카드/차트/후보군/최종 조합 후보)를 적용하지 않는다.

## 4. 타겟 유저 & 유즈케이스

- **캐주얼 로또 구매자**: 매주 습관적으로 로또를 사며, 재미로 번호를 다양하게 뽑아보고 싶어 함.
  - "여러 통계 전략 중 하나를 골라서 번호 조합을 받아보고 싶다."
  - "5만원어치 사려는데 몇 게임인지, 어떤 조합인지 한번에 보고 싶다."
  - "역대 가장 많이 나온 번호가 뭔지 궁금하다."
  - "내가 좋아하는 번호 몇 개는 고정하고 나머지만 뽑고 싶다."
  - "내 주변에서 1등이 잘 나온 판매점이 어딘지 알고 싶다."
- **가끔 구매자**: 판매점 위치를 몰라서 아무 데서나 사는 사람.
  - "지금 있는 곳 근처에 로또 판매점이 어디 있는지 지도로 보고 싶다."

## 5. 기술 스택 제안

AI 코딩 툴(Claude Code) 작업 효율과 무료 배포를 최우선으로 고려한 구성입니다.

| 영역 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | Next.js (App Router) + TypeScript | Claude Code가 다루기 쉽고, Vercel 무료 배포와 궁합이 좋음 |
| 스타일 | Tailwind CSS | 빠른 UI 작업, 별도 디자인 리소스 불필요 |
| 차트 | 커스텀 CSS/SVG 바 차트 | 활성도 그래프는 막대 45개 규모라 외부 차트 라이브러리 없이 충분 |
| 지도/위치 | Kakao Maps Web API (키워드/카테고리 장소 검색) | 무료 쿼터 제공, 국내 위치 데이터 정확도 높음 |
| 데이터 저장 | 정적 JSON 파일 (V1) → 추후 필요 시 Supabase/SQLite | 판매점·통계 데이터가 주 단위 갱신이라 DB 없이도 충분 |
| 번호 생성 로직 | 순수 TypeScript 함수 (규칙/통계 기반) | AI/LLM 불필요 — 13절 참고 |
| 배포 | Vercel (프론트) | 무료 티어로 충분, 별도 서버 운영 불필요 |
| 상태 저장 | localStorage | 로그인 없이 "최근 생성 번호", "즐겨찾기 판매점" 저장 |

## 6. 기능 요구사항

### F1. 번호 생성기 — 혼합 전략 + 분석 패널 (P0)

원자적 가중치 소스를 조합한 5가지 혼합 전략으로 번호를 생성하고, 결과를 다각도로 보여주는 분석 UI를 제공한다. 모바일 우선 반응형으로 구현한다.

**원자 가중치 소스**

| 소스 | 의미 | 계산 |
|---|---|---|
| 빈도(frequency) | 이력 전체 출현 횟수 | 번호별 전체 출현 횟수 집계 |
| 이월수(carryover) | 직전 회차 재출현 경향 | 직전 회차 번호 가중 부스트 |
| 미출현 역추세(cold) | 오래 안 나온 번호 | 마지막 출현 이후 경과 회차 수 |
| 이웃수(neighbor) | 인접 번호 경향 | 직전 회차 번호의 ±1~2 인접 번호 가중 |
| 동끝수(sameLastDigit) | 끝자리 동일 경향 | 직전 회차 번호와 1의 자리가 같은 번호 가중 |

**5가지 혼합 전략**

| 전략 id | 라벨 | 조합 | 설명 문구 |
|---|---|---|---|
| `frequency` | 빈도 기반 | freq×1 + carryover×1 + neighbor×1 + sameLastDigit×1 | "빈도 + 이월수 + 이웃수 + 동끝수" |
| `elite` | 엘리트 집중 | frequency×3 + carryover×3 + cold×2 | "빈도×3 + 이월수×3 + Cold×2" |
| `balanced` | 균형 조합 | 홀짝 3:3 + 구간 균등배분 | "홀짝 3:3 + 구간 균등배분" |
| `cold` | 미출현 역추세 | cold 상위 25수 풀 내 역추세 가중 랜덤 | "미출현 상위 25수 역추세 조합" |
| `random` | 종합 랜덤 | 위 4개 전략 가중치 평균 | "4개 전략 가중 평균 랜덤" |

**Acceptance Criteria**
- [ ] 사용자는 5가지 전략 중 하나를 선택할 수 있고, 각 전략은 아이콘·이름·조합 공식이 함께 표시된 카드 형태로 제공된다.
- [ ] 생성 세트 수를 1~5개 사이에서 조절할 수 있다.
- [ ] 제외 번호를 직접 입력하면 결과에서 제외된다.
- [ ] 포함(고정) 번호를 최대 5개까지 입력할 수 있다. 6개 전부를 고정하면 생성 자체가 무의미해지므로 5개로 제한하고, 제외 번호와 겹치면 경고 문구를 표시한다.
- [ ] 결과 화면에는 세트별 합계·홀짝 비율을 함께 표시한다.
- [ ] "적용 알고리즘 요약" 패널에 선택한 전략의 조합 공식을 사람이 읽을 수 있는 문장으로 보여준다.
- [ ] 1~45번 전체의 활성도(정규화된 가중치)를 막대그래프로 표시한다.
- [ ] "종합 랜덤" 전략을 선택했을 때만 후보군 패널을 추가로 보여준다: 1차 후보(빈도 기반 상위 12개) → 2차 후보(4개 전략 순위 합산 상위 12개) → 최종 조합 후보 8개.
- [ ] 최종 조합 후보 8개의 각 번호에는 실제 계산에서 나온 "특징 태그"만 붙인다(예: "이월수", "최근 24회 빈도 4회", "동끝수", 해당 없으면 "구간 균등 분포 후보"). **별점·순위·점수 등 정량적 우열 표시는 사용하지 않는다.** 패널 이름도 "AI 추천"이 아니라 "최종 조합 후보"처럼 중립적인 표현을 쓴다 — 이 앱은 실제 AI/LLM을 쓰지 않으므로(13절) "AI"라는 라벨은 없는 판단 근거가 있는 것처럼 보이게 하고, 별점은 통계적으로 무의미한 과거 빈도를 "더 나은 선택"처럼 보이게 해서 F5의 법적 원칙과 충돌한다.
- [ ] 번호 조합을 클립보드로 복사할 수 있다.
- [ ] 직전 회차 당첨번호를 상단에 표시한다(데이터 소스는 8절 참고).
- [ ] 결과 화면 하단에 상세 면책조항을 표시한다(문구는 F5 참고).
- [ ] 모바일(1열: 전략 카드 → 세트수/예산 → 제외·포함 번호 → 생성 버튼 → 결과 전체)과 데스크톱(`lg`, 1024px 기준 2열: 좌측 전략/입력, 우측 결과) 반응형 레이아웃을 적용한다.
- [ ] 활성도 차트, 특징 태그, 최종 조합 후보 카드에는 스크린리더용 대체 텍스트(aria-label)를 포함한다.

**기술 참고**
- 파일 구조(요약): `lib/lotto/weights.ts`(원자 가중치 5종), `lib/lotto/profiles.ts`(5개 혼합 전략 정의), `lib/lotto/analysis.ts`(특징 태그·후보군·최종 조합 후보 산출 로직), `components/lotto/`에 전략카드·결과요약·알고리즘요약패널·활성도차트·후보군리스트·최종조합후보카드·포함번호입력·상세면책조항 컴포넌트를 추가한다. 기존 `generate.ts`/`budget.ts`는 시그니처 변경 없이 그대로 재사용한다.
- `Strategy` 타입 값이 `frequency|carryover|balanced|cold|random`에서 `frequency|elite|balanced|cold|random`으로 바뀐다. F6(최근 생성 번호 히스토리)가 로컬에 전략 id를 저장하고 있었다면, 기존에 저장된 `carryover` 값을 새 UI에서 읽을 때 깨지지 않도록 마이그레이션(예: `carryover` → `frequency` 매핑) 또는 기본값 대체 처리를 반드시 포함한다.
- "종합 랜덤" 전략은 `elite` 자체가 이미 frequency·cold를 포함하므로, 4개 전략을 그대로 평균하면 frequency/cold가 사실상 이중 반영된다. 의도한 설계인지 구현 착수 전에 확인하고, 원치 않으면 원자 소스 5개를 균등 평균하는 방식으로 대체한다.
- 이 리치 UI(전략 카드, 활성도 차트, 후보군, 최종 조합 후보)는 F1(세트수 기반 생성)에만 적용한다. F2(예산 기반, 최대 200게임)는 게임 수가 많아 개별 후보·차트가 의미 없으므로 기존 단순 리스트 UI를 유지하고, 전략 계산 엔진(가중치 조합 로직)만 F1과 공유한다.
- 테스트: `weights.test.ts`(원자 소스 5종), `profiles.test.ts`(5개 전략이 유효한 6개 고유 번호를 반환하는지, 제외/포함 번호를 지키는지), `analysis.test.ts`(특징 태그 생성 조건, 후보군 12/12/8 산출 로직). UI는 Playwright로 모바일/데스크톱 뷰포트를 수동 검증한다.

### F2. 예산 기반 번호 생성 (P0)

로또 1게임(1줄)은 1,000원이다. 사용자가 구매 예정 금액을 정하면 그에 맞는 게임 수만큼 번호 조합을 자동 생성한다. F1의 전략 계산 엔진을 공유하되, UI는 리치 패널 없이 단순 리스트로 유지한다.

**Acceptance Criteria**
- [ ] 금액 프리셋 버튼 제공: 1,000 / 5,000 / 10,000 / 20,000 / 50,000원 + 직접 입력.
- [ ] 게임 수 = 입력 금액 ÷ 1,000원(소수점 이하 절사). 1,000원 단위가 아닌 금액 입력 시 안내 문구로 절사된 실제 게임 수를 보여준다.
- [ ] 생성되는 게임(조합)은 서로 중복되지 않도록 처리한다.
- [ ] F1의 5가지 전략 중 선택 적용 가능. 기본값은 "종합 랜덤".
- [ ] 결과 상단에 "총 OO원 · OO게임"을 요약 표시.
- [ ] 법적으로 오프라인 구매는 1인 1회 최대 20만원, 온라인(모바일/PC) 구매는 1인 1회 합산 5,000원으로 제한되어 있다(2026년 기준 시범 운영 규정). 이 앱은 실제 구매를 대행하지 않으므로 강제 상한은 20만원(200게임)으로만 두고, 5,000원을 초과해 선택할 경우 "온라인으로 직접 구매 시 1회 한도는 5,000원입니다"라는 참고 문구를 노출한다.
- [ ] 게임 수가 많을 경우(20게임 이상) 리스트가 길어지므로 접기/펼치기 UI를 적용한다.
- [ ] F1의 후보군/최종 조합 후보/활성도 차트 패널은 적용하지 않는다(200게임 단위에서 의미가 없음).

### F3. 역대 최다 출현 번호 TOP 6 (P0)

역대 전체 회차의 당첨번호를 집계해 가장 많이 나온 번호 순으로 최대 6개까지 보여준다.

**Acceptance Criteria**
- [ ] "역대 가장 많이 나온 번호는 O번입니다" 형태로 1위 번호를 강조 표시.
- [ ] 출현 빈도 순위 1~6위를 번호·출현 횟수와 함께 리스트로 표시.
- [ ] "펼쳐보기"로 전체 1~45번 순위를 확인할 수 있다(P1로 미뤄도 무방).
- [ ] 집계 기준 회차 범위와 마지막 갱신일을 화면에 명시("1회 ~ 1231회 기준").
- [ ] 이 통계는 과거 데이터일 뿐이며 향후 당첨을 예측하지 않는다는 디스클레이머를 함께 노출.
- [ ] F1의 `weights.ts` 빈도 집계 로직을 재사용해 중복 계산을 피한다.

**기술 참고**: 이 기능은 1회차부터 현재까지 전체 회차 데이터(1,200회 이상)가 필요하다. 매 요청마다 실시간으로 전 회차를 조회/집계하면 느리고 비효율적이므로, 빌드 타임 또는 주 1회 배치 작업으로 미리 집계한 정적 JSON(번호별 출현 횟수)을 만들어두고 프론트는 이 캐시 파일만 읽는 방식을 권장한다.

### F4. 판매점 랭킹 + 내 주변 찾기 (P0, 핵심 차별화 기능)

공공데이터(1등 배출 판매점 현황)를 기반으로 지역별/전국 랭킹을 보여주고, 사용자 위치 기준 가까운 순으로 정렬.

**Acceptance Criteria**
- [ ] 사용자가 위치 권한을 허용하면 현재 위치에서 가까운 순으로 판매점 목록을 정렬한다.
- [ ] 위치 권한 거부 시 지역(시/도) 선택 드롭다운으로 대체 조회가 가능하다.
- [ ] 각 판매점 카드에는 상호명, 주소, 1등/2등 배출 횟수, 거리(km)를 표시한다.
- [ ] 지도 화면에서 마커 클릭 시 상세 정보와 길찾기 링크(카카오맵 연결)를 제공한다.
- [ ] 데이터 최종 업데이트 일자를 화면에 명시한다("이 정보는 YYYY-MM-DD 기준입니다").

### F5. 법적 안전장치 (P0, 전 화면 공통)

**Acceptance Criteria**
- [ ] 앱 최초 진입 시 1회 안내 문구 노출: "이 앱은 재미를 위한 서비스이며 당첨을 보장하지 않습니다."
- [ ] 하단 푸터에 동일 문구 상시 노출.
- [ ] 결제/구독/환불 관련 UI 요소를 전혀 포함하지 않는다.
- [ ] F1 결과 화면에는 아래 상세 면책조항을 표시한다(기준 회차·날짜는 최신 당첨번호 조회 응답값을 그대로 사용):

  > 본 분석은 통계 기반 참고 정보입니다(기준 회차: OO회, YYYY-MM-DD). 실제 결과와 차이가 있을 수 있으며 당첨을 보장하지 않습니다. 로또는 1~45 중 6개를 무작위로 추첨하는 완전 확률 게임이며, 과거 패턴이 미래 결과를 예측하지 않습니다. 본 정보는 재미를 위한 통계이며 투자·도박 조언이 아닙니다. 정확한 당첨 결과는 동행복권(dhlottery.co.kr)에서 확인하세요.

- [ ] 번호별 특징 태그, 후보군, 최종 조합 후보를 포함해 앱 전체 어디에도 별점·순위·점수·"추천"·"AI"라는 표현을 사용하지 않는다. 시각적으로 확신을 주는 등급 표시(색상 강조 정도는 허용, 등급/랭킹 뱃지는 금지)를 사용하지 않는다.

### F6 (P1). 최근 생성 번호 히스토리

1. **홈**: 직전 회차 당첨번호 + 번호 생성기 진입 + 예산 기반 생성 진입 + 최다출현 TOP 6 미리보기 + 판매점 찾기 진입

## 13. AI/RAG 사용 여부에 대한 의견

**현재 설계는 AI(LLM)를 쓰지 않는 게 맞다.** F1~F3의 번호 생성/통계 로직은 과거 당첨 데이터를 집계하고 가중치를 둬서 조합을 뽑는 순수 통계·규칙 기반 계산이다. 참고 사이트의 "AI 종합" 전략도 실제로는 기존 전략들을 섞어 가중 랜덤으로 뽑는 것뿐, 언어모델이 관여하지 않는다. 이건 숨기거나 아쉬워할 부분이 아니라 오히려 맞는 설계다 — 로또는 완전 무작위이므로 계산은 빠르고 정확한 결정론적 로직이 LLM보다 훨씬 낫다.

**RAG는 이 문제에 맞는 도구가 아니다.** RAG는 비정형 문서를 검색해서 언어모델이 답변을 생성할 때 근거를 붙여주는 기술이다. 여기서 다루는 데이터는 회차·번호·판매점처럼 완전히 정형화된 숫자 데이터라, 벡터 검색보다 DB 조회/집계 쿼리가 훨씬 빠르고 정확하다.

**F1 리치 UI 설계에도 이 원칙이 그대로 적용된다.** "최종 조합 후보" 패널은 순수 통계 계산 결과만 보여주고, 별점이나 "AI 추천" 같은 라벨은 사용하지 않는다(F1/F5 참고). "AI"라는 단어를 UI에 쓰고 싶다면, 번호 생성 로직은 그대로 두고 결과를 설명하는 코멘트에만 LLM을 붙이는 방향(F8, V2 이후)을 권장한다.
