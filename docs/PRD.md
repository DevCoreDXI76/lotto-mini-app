# 로또 미니앱 — 개발 스펙 (PRD)

> 이 문서는 VSCode의 Claude Code(또는 다른 AI 코딩 에이전트)에 그대로 전달해서 개발을 진행하기 위한 문서입니다. 비개발자가 AI 코딩 툴로 작업한다는 전제로 작성했습니다.
>
> **갱신 이력(F4 데이터 조사 반영)**: F4 설계 전 실제 공공데이터를 조사한 결과, 기존 가정("판매점 데이터가 주 단위 갱신")이 틀렸고 좌표 미포함·매칭 이슈 등 제약이 확인되어 F4 Acceptance Criteria, Non-goals, 데이터 소스(9절), Open Questions, 타임라인을 실제 데이터 현실에 맞게 다시 정리했다.

## 0. 로드맵 — 1단계 MVP / 2단계 고도화

**1단계(MVP)**: 아래 F1~F9(리치 UI 포함 F1 그대로 유지)를 완성해 **텔레그램 미니앱으로 먼저 출시**한다. 목표는 완벽함이 아니라 "번호 생성기 + 예산 기반 생성 + 판매점 찾기"라는 핵심 가설을 실제 사용자에게 검증받는 것이다. 카카오 채널 준비는 별도 트랙으로 병행하되 텔레그램 출시를 막지 않는다.

**2단계(고도화)**: 1단계 출시 후 모은 피드백을 반영해 F10~F12(구매 체크인, 당첨 확인 알림, 위치 기반 배너)를 추가한다. 각 기능은 "사용자가 로또를 아예 잊고 못 사는 것"과 "당첨금을 놓치는 것"이라는 두 가지 실제 문제를 겨냥한다 — 자세한 배경은 F10~F12 참고.

**카카오 트랙에 대한 중요한 정정**: 이전에 "카카오 미니앱은 비즈니스 심사와 SDK 연동이 필요하다"고 언급했는데, 이번에 다시 확인해보니 개발자가 자유롭게 앱을 등록할 수 있는 공개된 "카카오 미니앱 플랫폼"은 확인되지 않았다. 검색으로 찾은 것은 카카오톡 게임플레이·카카오페이 미니게임처럼 카카오가 직접 큐레이션하는 퍼스트파티/제휴 프로그램뿐이었다(2025년 11월경 시작). 즉, 지금 시점에 솔로 개발자가 자기 앱을 "카카오 미니앱"으로 셀프 등록할 수 있는 공식 채널은 확인되지 않는다. 대신 실제로 확인된, 지금 바로 쓸 수 있는 경로는 **카카오톡 채널(챗봇) 개설 + 웹앱 링크 연결**이다 — 카카오톡은 채팅 중 공유된 웹 링크를 인앱 브라우저(내장 웹뷰)로 열기 때문에, 채널을 만들고 그 안에 웹앱 링크를 걸어두면 사용자 입장에서는 카카오톡을 벗어나지 않고 앱을 쓰는 경험을 얻을 수 있다. "카카오 미니앱"이라는 공식 명칭의 기능은 아니지만 실질적으로 비슷한 효과를 낸다. 자세한 준비 항목은 F13 참고.

## 1. 문제 정의

로또 번호 생성기 앱은 이미 시장에 많지만, 대부분 "번호 생성" 기능 하나에만 집중되어 있고 실제로 도움이 되는 정보(가까운 고배출 판매점)는 잘 노출되지 않는다. 사용자는 재미로 다양한 방식의 번호를 뽑아보고 싶어 하면서, 동시에 "어디서 사야 하는지"에 대한 실용적인 궁금증도 갖고 있다. 이 둘을 하나의 가벼운 유틸리티 앱으로 묶는다.

**중요한 법적 전제**: 이 앱은 당첨 확률을 높여준다고 주장하지 않는다. 번호 생성은 재미/통계적 흥미 요소이고, 모든 화면에 "당첨을 보장하지 않습니다" 류의 디스클레이머를 노출한다. 유료 결제·환불 보장 문구는 사용하지 않는다(무료 서비스). 번호별로 우열을 암시하는 표현(별점, 순위, "추천" 등)도 사용하지 않는다 — 자세한 원칙은 F1/F5 참고.

## 2. 목표

- 1단계(MVP)를 6주 이내(주당 약 21시간 투입 기준)에 텔레그램 미니앱으로 출시하고 실사용자 트래픽을 받는다.
- "가까운 판매점 찾기" 기능의 실사용률(방문자 대비 위치 조회 클릭 비율) 30% 이상을 1차 성공 기준으로 삼는다.
- 번호 생성기 재사용률(같은 세션에서 2회 이상 생성) 20% 이상.
- 출시 2주 내 피드백(F9) 최소 10건 이상 수집해 2단계 우선순위 결정에 반영한다.
- 법적 리스크 없이(과장 광고 문구, 결제 유도, 번호별 우열 암시, 과도한 구매 유도 없이) 운영한다.

## 3. Non-goals (1단계 범위 제외)

- 회원가입/로그인 — 익명 사용, 즐겨찾기 등은 로컬 저장(localStorage)으로 대체.
- 유료 결제, 프리미엄 기능 — 법적 리스크 회피 및 속도를 위해 완전 무료로 시작.
- 카카오 공식 미니앱 SDK 연동 — 0절 참고, 현재 자기등록 가능한 공개 플랫폼이 확인되지 않아 1단계에서는 카카오톡 채널+웹뷰 링크(F13)로 대체.
- **전국 판매점 실시간 동기화 — 애초에 불가능한 정도가 아니라, 원천 데이터 자체가 그럴 수 없는 구조다.** 1등 배출 판매점 데이터는 연 1회 갱신, 전체 판매점 주소 데이터는 2025-06-07 기준 1회성 스냅샷이다(9절 참고). "동기화"라는 표현 자체가 이 데이터 성격과 안 맞으므로, 정기적으로(예: 반기 1회) 원본 데이터셋이 새로 올라왔는지 수동 확인 후 재적재하는 방식으로 운영한다.
- 번호 자동 구매/연동 — 동행복권 구매 대행은 법적으로 별도 이슈이므로 제외. "번호 복사하기" 정도만 지원.
- LLM 기반 번호 생성 — 13절 참고. 번호 생성 로직은 규칙 기반 통계로만 구현한다.
- 참고 사이트(e-ubis.com 등)의 정확한 가중치 배율을 그대로 복제하지 않는다 — 조합 비율은 이 문서(F1)에 정의된 값을 따른다.
- 근거/특징 문구를 그럴듯하게 지어내지 않는다 — 계산 불가능한 항목은 표시하지 않는다. **F4의 1등 배출 배지도 동일 원칙 — 상호명·지역 매칭이 애매한 경우 특정 매장에 단정적으로 배지를 붙이지 않는다(자세한 내용은 F4 참고).**
- 번호별 별점·순위·점수 같은 정량적 우열 표시는 사용하지 않는다 — 이유는 F1 참고.
- F2(예산 기반)·F3(역대 통계)·F4(판매점 찾기)에는 F1의 리치 UI(전략카드/차트/후보군/최종 조합 후보)를 적용하지 않는다.
- F10~F12(구매 체크인, 당첨 확인 알림, 위치 기반 배너)는 2단계로 미룬다 — 1단계에는 포함하지 않는다.
- 백그라운드 위치 추적(지나가면 자동 알림) — 현재 웹앱(PWA) 구조로는 iOS 등에서 사실상 구현이 불가능해 시도하지 않는다. F12는 포그라운드(앱을 열었을 때)에서만 동작한다.
- 하루에 여러 번 구매를 재촉하는 알림 — 사행성 조장 우려로 시도하지 않는다. F10은 옵트인·저빈도로만 설계한다.

## 4. 타겟 유저 & 유즈케이스

- **캐주얼 로또 구매자**: 매주 습관적으로 로또를 사며, 재미로 번호를 다양하게 뽑아보고 싶어 함.
  - "여러 통계 전략 중 하나를 골라서 번호 조합을 받아보고 싶다."
  - "5만원어치 사려는데 몇 게임인지, 어떤 조합인지 한번에 보고 싶다."
  - "역대 가장 많이 나온 번호가 뭔지 궁금하다."
  - "내가 좋아하는 번호 몇 개는 고정하고 나머지만 뽑고 싶다."
  - "내 주변에 있는 판매점을 다 보고 싶고, 그중에 1등 나온 곳이 어딘지도 알고 싶다."
- **가끔 구매자**: 판매점 위치를 몰라서 아무 데서나 사는 사람.
  - "지금 있는 곳 근처에 로또 판매점이 어디 있는지 지도로 보고 싶다."
- **깜빡하는 구매자(2단계 타깃)**: "이번 주엔 꼭 사야지" 하고 잊어버리는 사람, 사놓고 당첨 확인을 놓치는 사람.
  - "이번 주 아직 안 샀으면 살짝 알려줬으면 좋겠다."
  - "산 번호를 기억해뒀다가 당첨됐는지 자동으로 알려줬으면 좋겠다."

## 5. 기술 스택 제안

AI 코딩 툴(Claude Code) 작업 효율과 무료 배포를 최우선으로 고려한 구성입니다.

| 영역 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | Next.js (App Router) + TypeScript | Claude Code가 다루기 쉽고, Vercel 무료 배포와 궁합이 좋음 |
| 스타일 | Tailwind CSS | 빠른 UI 작업, 별도 디자인 리소스 불필요 |
| 차트 | 커스텀 CSS/SVG 바 차트 | 활성도 그래프는 막대 45개 규모라 외부 차트 라이브러리 없이 충분 |
| 지도/위치 | Kakao Maps Web API (지도 표시, 주소→좌표 변환, 마커 클러스터러) | 무료 쿼터 제공(단, 지오코딩은 쿼터 재확인 필요 — 9절 참고) |
| 1차 배포 채널 | 텔레그램 미니앱(웹뷰) | 자체 웹앱을 그대로 감싸는 수준이라 진입장벽이 낮음 |
| 2차 배포 채널 | 카카오톡 채널 + 웹뷰 링크 | F13 참고, 공식 미니앱 SDK 대신 채널 기반 접근 |
| 데이터 저장 | 정적 JSON 파일 (1단계) → 추후 필요 시 Supabase/SQLite | 판매점·통계 데이터가 주 단위 갱신이라 DB 없이도 충분 |
| 번호 생성 로직 | 순수 TypeScript 함수 (규칙/통계 기반) | AI/LLM 불필요 — 13절 참고 |
| 배포 | Vercel (프론트) | 무료 티어로 충분, 별도 서버 운영 불필요 |
| 상태 저장 | localStorage | 로그인 없이 "최근 생성 번호", "즐겨찾기 판매점" 저장 |
| 피드백/분석 | 텔레그램 봇 메시지 forward + 구글폼 + Vercel Analytics | F9 참고, 전부 무료 티어로 구성 |

## 6. 기능 요구사항 — 1단계(MVP)

### F1. 번호 생성기 — 혼합 전략 + 분석 패널 (P0)

원자적 가중치 소스를 조합한 5가지 혼합 전략으로 번호를 생성하고, 결과를 다각도로 보여주는 분석 UI를 제공한다. 모바일 우선 반응형으로 구현한다. (사용자 결정: 리치 UI는 축소하지 않고 1단계에 그대로 포함한다.)

**원자 가중치 소스**

| 소스 | 의미 | 계산 |
|---|---|---|
| 빈도(frequency) | 이력 전체 출현 횟수 | 번호별 전체 출현 횟수 집계 |
| 이월수(carryover) | 직전 회차 재출현 경향 | 직전 회차 번호 가중 부스트 |
| 미출현 역추세(cold) | 오래 안 나온 번호 | 마지막 출현 이후 경과 회차 수 |
| 이웃수(neighbor) | 인접 번호 경향 | 직전 회차 번호의 ±1~2 인접 번호 가중 |
| 동끝수(sameLastDigit) | 끝자리 동일 경향 | 직전 회차 번호와 1의 자리가 같은 번호 가중 |

**5가지 혼합 전략**

| 전략 id | 라벨 | 조합 | 설명 문구 |
|---|---|---|---|
| `frequency` | 빈도 기반 | freq×1 + carryover×1 + neighbor×1 + sameLastDigit×1 | "빈도 + 이월수 + 이웃수 + 동끝수" |
| `elite` | 엘리트 집중 | frequency×3 + carryover×3 + cold×2 | "빈도×3 + 이월수×3 + Cold×2" |
| `balanced` | 균형 조합 | 홀짝 3:3 + 구간 균등배분 | "홀짝 3:3 + 구간 균등배분" |
| `cold` | 미출현 역추세 | cold 상위 25수 풀 내 역추세 가중 랜덤 | "미출현 상위 25수 역추세 조합" |
| `random` | 종합 랜덤 | 위 4개 전략 가중치 평균 | "4개 전략 가중 평균 랜덤" |

**Acceptance Criteria**
- [ ] 사용자는 5가지 전략 중 하나를 선택할 수 있고, 각 전략은 아이콘·이름·조합 공식이 함께 표시된 카드 형태로 제공된다.
- [ ] 생성 세트 수를 1~5개 사이에서 조절할 수 있다.
- [ ] 제외 번호를 직접 입력하면 결과에서 제외된다.
- [ ] 포함(고정) 번호를 최대 5개까지 입력할 수 있다. 6개 전부를 고정하면 생성 자체가 무의미해지므로 5개로 제한하고, 제외 번호와 겹치면 경고 문구를 표시한다.
- [ ] 결과 화면에는 세트별 합계·홀짝 비율을 함께 표시한다.
- [ ] "적용 알고리즘 요약" 패널에 선택한 전략의 조합 공식을 사람이 읽을 수 있는 문장으로 보여준다.
- [ ] 1~45번 전체의 활성도(정규화된 가중치)를 막대그래프로 표시한다.
- [ ] "종합 랜덤" 전략을 선택했을 때만 후보군 패널을 추가로 보여준다: 1차 후보(빈도 기반 상위 12개) → 2차 후보(4개 전략 순위 합산 상위 12개) → 최종 조합 후보 8개.
- [ ] 최종 조합 후보 8개의 각 번호에는 실제 계산에서 나온 "특징 태그"만 붙인다(예: "이월수", "최근 24회 빈도 4회", "동끝수", 해당 없으면 "구간 균등 분포 후보"). **별점·순위·점수 등 정량적 우열 표시는 사용하지 않는다.** 패널 이름도 "AI 추천"이 아니라 "최종 조합 후보"처럼 중립적인 표현을 쓴다.
- [ ] 번호 조합을 클립보드로 복사할 수 있다.
- [ ] 직전 회차 당첨번호를 상단에 표시한다(데이터 소스는 8절 참고).
- [ ] 결과 화면 하단에 상세 면책조항을 표시한다(문구는 F5 참고).
- [ ] 모바일(1열: 전략 카드 → 세트수/예산 → 제외·포함 번호 → 생성 버튼 → 결과 전체)과 데스크톱(`lg`, 1024px 기준 2열: 좌측 전략/입력, 우측 결과) 반응형 레이아웃을 적용한다.
- [ ] 활성도 차트, 특징 태그, 최종 조합 후보 카드에는 스크린리더용 대체 텍스트(aria-label)를 포함한다.

**기술 참고**
- 파일 구조(요약): `lib/lotto/weights.ts`(원자 가중치 5종), `lib/lotto/profiles.ts`(5개 혼합 전략 정의), `lib/lotto/analysis.ts`(특징 태그·후보군·최종 조합 후보 산출 로직), `components/lotto/`에 전략카드·결과요약·알고리즘요약패널·활성도차트·후보군리스트·최종조합후보카드·포함번호입력·상세면책조항 컴포넌트를 추가한다.
- `Strategy` 타입 값이 `frequency|carryover|balanced|cold|random`에서 `frequency|elite|balanced|cold|random`으로 바뀐다. F6(최근 생성 번호 히스토리)가 로컬에 전략 id를 저장하고 있었다면 마이그레이션(예: `carryover` → `frequency` 매핑) 또는 기본값 대체 처리를 반드시 포함한다.
- "종합 랜덤" 전략은 `elite` 자체가 이미 frequency·cold를 포함하므로, 4개 전략을 그대로 평균하면 frequency/cold가 사실상 이중 반영된다. 의도한 설계인지 구현 착수 전에 확인하고, 원치 않으면 원자 소스 5개를 균등 평균하는 방식으로 대체한다.
- 테스트: `weights.test.ts`, `profiles.test.ts`, `analysis.test.ts`. UI는 Playwright로 모바일/데스크톱 뷰포트를 수동 검증한다.

### F4. 판매점 지도 — 전체 표시 + 1등 배출 이력 배지 (P0, 핵심 차별화 기능)

**설계 원칙 변경(F4 데이터 조사 반영)**: 처음엔 "1등 배출 판매점만" 보여주는 걸로 설계했으나, 실제 데이터 조사 결과를 반영해 **전체 판매점을 기본으로 보여주고, 그중 1등 배출 이력이 확인되는 곳만 배지를 추가로 붙이는 구조**로 변경한다. 두 데이터셋 다 좌표가 없어 지오코딩이 선행되어야 하고, 두 데이터셋 간 조인은 완벽하지 않다는 걸 전제로 설계한다.

**데이터 파이프라인 (배치, 빌드 타임 또는 수동 실행)**
1. 전체 판매점 주소 데이터(8,802건, 9절 데이터셋 B)의 도로명주소를 Kakao 주소검색 API로 지오코딩한다. 도로명주소 변환 실패 시 지번주소로 재시도한다. 실패한 항목은 별도 리스트로 남기고(좌표 없이 목록에서만 노출하거나 제외), 실패율을 로그로 남긴다.
2. 1등 배출 판매점 데이터(463건, 데이터셋 A)를 상호명 + 지역(시군구) 기준으로 위 지오코딩된 전체 판매점 리스트와 매칭한다.
   - 상호명+시군구로 후보가 정확히 1개면 그 판매점에 1등 배출 건수를 배지로 붙인다.
   - 후보가 2개 이상(동일 상호명이 같은 시군구에 여럿)이면 **특정 매장에 배지를 단정적으로 붙이지 않는다.** 대신 해당 시군구 결과 목록 상단에 "이 지역 'OO' 계열 매장 중 1곳에서 1등 배출 이력이 있습니다(정확한 지점 확인 안 됨)"처럼 모호성을 그대로 드러내는 방식으로 처리하거나, 1단계에서는 배지 없이 매칭 실패 목록으로 남겨두고 2단계에서 정밀 매칭(예: 반경 검증)을 검토한다.
   - 매칭 결과(성공/모호/실패 건수)를 빌드 로그에 남긴다.
3. 위 결과를 정적 JSON으로 저장하고 프론트는 이 캐시만 읽는다.

**Acceptance Criteria**
- [ ] 지도에는 전국 판매점 전체(약 8,802건, 지오코딩 성공분)가 표시된다.
- [ ] 1등 배출 이력이 확실하게 매칭된 판매점에는 "1등 O회 배출" 배지가 표시된다. **이 배지는 자동선택 당첨 기준이며, 수동선택 당첨은 집계에 포함되지 않는다는 점을 배지 옆 또는 상세 화면에 명시한다.**
- [ ] 매칭이 모호한 경우 특정 매장에 배지를 붙이지 않고, 위 파이프라인 3번 원칙대로 처리한다.
- [ ] 위치 권한 허용 시 현재 위치 기준 가까운 순 정렬. 거부 시 지역(시/도) 선택 드롭다운으로 대체.
- [ ] 각 판매점 카드에 상호명, 주소, (있으면) 1등 배출 횟수, 거리(km) 표시.
- [ ] 지도 화면에서 마커 클릭 시 상세 정보와 길찾기 링크 제공.
- [ ] **마커 클러스터링을 필수로 적용한다**(Kakao Maps MarkerClusterer). 8,802개를 클러스터링 없이 렌더링하면 전국 단위 줌에서 성능·가독성 문제가 발생하므로 선택 사항이 아니라 필수 요구사항으로 취급한다.
- [ ] 데이터 기준일을 화면에 명시한다: "판매점 목록은 2025-06-07 기준이며, 폐업·신규 매장이 반영되지 않았을 수 있습니다. 1등 배출 이력은 [최신 갱신 연도] 기준입니다."

### F10. 구매 리마인더 / 상태 기반 체크인 (P1)

사용자가 "이번 주엔 꼭 사야지" 하고 잊어버리는 문제를 겨냥한다. 문자 그대로 매일 반복하지 않고, 이번 회차에 대해 아직 응답하지 않은 사용자에게만 하루 한 번 정도 묻는 상태 기반 설계로 한다.

**설계 원칙**
- [ ] 기본값은 꺼짐(옵트인) — 사용자가 명시적으로 켜야 알림/체크인이 시작된다.
- [ ] 이번 주 회차에 대해 이미 "샀어요" 응답을 남긴 사용자에게는 같은 회차에 대해 다시 묻지 않는다.
- [ ] 문구는 항상 담담한 정보 전달 톤을 쓴다(F5 참고) — 구매를 재촉하는 인상을 주지 않는다.
- [ ] "샀어요" 응답과 함께 남긴 번호 조합은 F11(당첨 확인 알림)의 데이터 소스로 재사용한다.
- [ ] 텔레그램 채널 기준으로는 봇 메시지로, 웹앱 기준으로는 Web Push(가능한 브라우저 한정, iOS Safari 등 미지원 환경에서는 앱 오픈 시 배너로 대체)로 구현한다.

### F11. 당첨 확인 자동 알림 (P1)

사용자가 산 번호를 기억해뒀다가 매주 추첨 후 자동으로 당첨 여부를 확인해서 알려준다. 로또 당첨금 중 상당액이 매년 수령 시효(1년)를 넘겨 소멸된다는 문제의식에서 출발한 기능으로, 재미가 아니라 실용적 도움에 가깝다.

**설계 원칙**
- [ ] F10에서 기록된(또는 F1/F2에서 생성 후 사용자가 직접 "구매완료" 표시한) 번호만 대상으로 한다 — 생성만 하고 실제로 안 산 번호까지 확인 대상에 넣지 않는다.
- [ ] 추첨 결과 발표 후(토요일 저녁) 자동으로 대조해 몇 개 맞았는지 알려준다.
- [ ] 5등 이상 당첨 시에는 "동행복권 홈페이지 또는 판매점에서 수령 방법을 확인하세요"라는 안내와 함께, 수령 시효(1년)가 있다는 점을 명시한다.

### F12. 즐겨찾기 판매점 근처 배너 (P1)

"지나가면 자동 알림"은 백그라운드 위치 추적이 필요해 현재 웹앱 구조로는 구현이 어렵다(3절 Non-goals 참고). 대신 앱을 열었을 때(포그라운드)만 동작하는 가벼운 버전으로 구현한다.

**설계 원칙**
- [ ] 앱을 열 때 이미 보유한 위치 권한으로 현재 위치와 F7 즐겨찾기 판매점 목록을 대조한다.
- [ ] 즐겨찾기 판매점과 가까운 거리(기준 거리는 구현 시 확정, 예: 도보 5분 내외)에 있고 이번 주 아직 F10 체크인을 안 한 경우에만 배너를 노출한다.
- [ ] 백그라운드 알림이 아니므로 "지나갈 때마다"는 아니고 "앱을 켰을 때"로 기대치를 명확히 안내한다.

### F13. 카카오톡 채널 준비 트랙 (P1, 텔레그램 출시와 병행, 출시를 막지 않음)

**준비 항목**

## 14. AI/RAG 사용 여부에 대한 의견

**현재 설계는 AI(LLM)를 쓰지 않는 게 맞다.** F1~F3의 번호 생성/통계 로직은 과거 당첨 데이터를 집계하고 가중치를 둬서 조합을 뽑는 순수 통계·규칙 기반 계산이다. 로또는 완전 무작위이므로 계산은 빠르고 정확한 결정론적 로직이 LLM보다 훨씬 낫다.

**RAG는 이 문제에 맞는 도구가 아니다.** RAG는 비정형 문서를 검색해서 언어모델이 답변을 생성할 때 근거를 붙여주는 기술이다. 여기서 다루는 데이터는 회차·번호·판매점처럼 완전히 정형화된 숫자 데이터라, 벡터 검색보다 DB 조회/집계 쿼리가 훨씬 빠르고 정확하다.

**F1 리치 UI 설계에도, 2단계 F10~F12에도 이 원칙이 그대로 적용된다.** "최종 조합 후보"·구매 체크인·당첨 확인 알림 모두 순수 계산/데이터 대조 결과만 전달하고, 별점이나 "AI 추천" 같은 라벨은 사용하지 않는다. "AI"라는 단어를 UI에 쓰고 싶다면, 번호 생성 로직은 그대로 두고 결과를 설명하는 코멘트에만 LLM을 붙이는 방향(F8, 장기 고려)을 권장한다.
