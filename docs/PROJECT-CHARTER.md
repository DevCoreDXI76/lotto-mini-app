# 프로젝트 확장 기획서 — 로또 미니앱 → 도구형 수익화 웹사이트

> 이 문서는 `docs/PRD.md`(1단계 MVP 스펙)와 `docs/ARCHITECTURE.md`(기술 설계)를 검토한 뒤 작성한 확장 단계 기획 문서다. 기존 두 문서를 대체하지 않고, 그 위에 "수익화를 목표로 한 도구형 웹사이트 확대"라는 새 트랙을 얹는다. 3가지 확인 질문(수익화 방식/서비스 범위/네이밍)에 대한 답을 반영했다: **광고 중심 수익화**, **서비스 범위는 문서 내 단계별 옵션으로 제시**, **네이밍은 기존 브랜드 확장안과 신규 브랜드안을 둘 다 제시**.

## 0. 현재 상태 요약 (근거: 첨부 폴더)

- 현재 제품명은 "로또 미니앱", 텔레그램 봇 `@lotto_mini_bot`으로 배포된 MVP. 도메인은 `lotto-mini-app-three.vercel.app`(Vercel 무료 티어).
- 구현 완료: F1(혼합 전략 번호 생성기), F2(예산 기반 생성), F3(역대 통계), F4(전국 판매점 지도 + 1등 배출 배지), F5(법적 안전장치), F6~F9(히스토리/즐겨찾기/피드백).
- PRD 3절 Non-goals에 "유료 결제·프리미엄 기능 없음", "완전 무료로 시작"이 명시돼 있고, F5(법적 안전장치)는 앱 전체에 걸쳐 당첨 보장 암시·별점/순위/추천 표현·결제 유도 문구를 금지한다.
- PRD 11절 Open Question에 이미 "무료 서비스라도 광고 수익화를 고려한다면 광고 문구 심의 기준을 별도로 검토 필요"라는 미해결 항목이 있다 — 이 기획서가 그 항목에 대한 답이자 실행 계획이다.
- 2단계 기능(F10 구매 체크인, F11 당첨 확인 알림, F12 위치 기반 배너)은 아직 미착수 상태로 남아 있다.

**중요한 전제 변경**: 기존 PRD는 "완전 무료·수익화 없음"을 전제로 법적 리스크를 회피했다. 이 문서는 그 전제 중 "수익화 없음"만 수정하고, "당첨 확률 과장 금지·결제(=복권 판매) 없음·우열 표현 금지"라는 나머지 법적 안전장치는 그대로 유지한 채 광고 수익을 얹는 방식을 제안한다.

## 1. 프로젝트 이름

두 가지 방향을 비교표로 제시한다. 최종 선택은 사용자가 한다.

### 옵션 A — 기존 브랜드 확장 (추천)

기존 자산(텔레그램 봇 `@lotto_mini_bot`, 도메인, 카카오 채널 문구, "로또 미니앱" 검색 노출)을 그대로 승계하면서 "도구 모음" 느낌을 더하는 이름.

| 후보 | 컨셉 | 비고 |
|---|---|---|
| **로또랩 (LottoLab)** | "번호 생성기 + 통계 + 판매점 찾기"를 실험/분석하는 연구소 이미지 | 도구가 여러 개 모인 사이트라는 확장 의도를 잘 드러냄 |
| **로또허브 (LottoHub)** | 여러 로또 관련 도구를 한곳에 모은 허브 | 2단계에서 연금복권 등으로 넓힐 때도 "허브"라는 이름이 자연스럽게 확장됨 |
| 로또플러스 (Lotto Plus) | 기존 이름 + 확장 뉘앙스 | 흔히 쓰이는 이름이라 상표·검색 변별력이 약함(참고 사이트가 이미 다수 존재) |

**장점**: "로또 번호 생성기", "로또 판매점" 같은 핵심 검색어에 브랜드명 자체가 이미 대응돼 SEO 초기 진입 장벽이 낮다. 광고 수익 모델은 검색 트래픽이 생명줄이므로 이 이점이 크다. 기존 봇 이름·채널 문구·사용자 기억을 재사용할 수 있어 전환 비용이 적다.

**단점**: "로또"라는 단어가 브랜드명에 고정되면, 2단계에서 연금복권·스피또 등으로 영역을 넓힐 때 이름이 다소 좁게 느껴질 수 있다(다만 "로또"가 한국에서는 복권 전체를 가리키는 관용어처럼 쓰이는 경우도 많아 치명적이진 않음). 광고 심사 관점에서 "로또"가 도메인/브랜드명에 그대로 들어가면 일부 광고 네트워크(2절 참고)가 사행산업 관련 카테고리로 자동 분류할 가능성도 있다.

### 옵션 B — 신규 브랜드

"로또"라는 단어를 브랜드명에서 빼고, 추첨(draw)/통계(stat) 자체를 상징하는 이름으로 재출발.

| 후보 | 컨셉 | 비고 |
|---|---|---|
| **드로우랩 (DrawLab)** | "추첨(draw)"을 다루는 도구 연구소 | 로또뿐 아니라 연금복권·스피또 등 다른 추첨형 복권도 자연스럽게 포괄 |
| 넘랩 (NumLab) | 숫자(number) 기반 도구 | 로또와의 연관성이 이름만으로는 잘 안 보여 첫 진입 시 설명이 더 필요함 |
| 픽스탯 (PickStat) | 번호 선택(pick) + 통계(stat) | 통계 도구라는 색이 강해 판매점 찾기 등 비통계 기능과는 다소 안 맞음 |

**장점**: 브랜드가 특정 게임(로또 6/45)에 종속되지 않아 2단계 확장이 더 자유롭다. "로또"라는 단어가 빠지면 광고 네트워크의 사행산업 카테고리 자동 분류 리스크도 줄어들 가능성이 있다(단, 콘텐츠 자체가 로또를 다루므로 완전히 회피되진 않는다 — 2절 참고).

**단점**: 처음부터 브랜드 인지도를 새로 쌓아야 하고, 기존 텔레그램 봇·채널 자산과 이름이 어긋나 사용자 혼란(리브랜딩 공지, 리다이렉트 등)이 생긴다. 핵심 검색어("로또 번호 생성기")에 브랜드명이 직접 대응하지 않아 SEO 초기 속도가 옵션 A보다 느릴 수 있다.

### 추천

**광고 수익화가 목표라면 트래픽(SEO)이 가장 중요한 변수이므로, 옵션 A(로또랩 또는 로또허브)를 1차 추천한다.** 2단계에서 실제로 다른 복권 종류까지 다룰 확정 계획이 서면 그때 "허브"라는 이름 아래 하위 서비스를 늘리는 방식이 리브랜딩보다 비용이 적다. 최종 이름 선택 전에 도메인(`.com`/`.co.kr`)과 상표 충돌 여부를 별도로 확인해야 한다(이 문서에서는 확인하지 않았다).

## 2. 프로젝트 목표

### 비전

"로또/복권 이용자가 필요로 하는 무료 도구(번호 생성·통계·판매점 찾기)를 한곳에 모은 사이트를 만들어, 방문자 트래픽을 발판으로 광고 수익을 안정적으로 창출한다." 기존 F5의 법적 안전장치(당첨 보장 안 함, 우열 표현 금지)는 브랜드 신뢰의 핵심 자산으로 그대로 유지한다 — "과장하지 않는 담백한 도구"라는 포지셔닝 자체가 장기적으로 재방문·검색 신뢰도에 도움이 된다.

### 1단계 목표 (수익화 인프라 구축, 착수 후 약 2~3개월)

- 텔레그램 미니앱에 갇혀 있던 서비스를 **독립 웹사이트로 SEO 진입 가능하게 전환**한다 — 텔레그램은 유지하되 "재방문·알림 채널"로 역할을 재정의하고, 웹(검색 유입)을 1차 획득 채널로 삼는다.
- **Google AdSense 심사를 통과하고 광고 게재를 시작**한다(2절 참고 — 순수 유틸리티 성격의 로또 번호 생성기 사이트/앱이 실제로 광고를 게재하는 선례가 다수 확인됨).
- 핵심 검색어(예: "로또 번호 생성기", "로또 판매점 찾기", "로또 당첨번호 통계") 기준 검색 노출을 확보하기 위한 SEO 구조(메타데이터, 사이트맵, 개별 랜딩 페이지)를 갖춘다.
- 광고 게재 후 첫 1~2개월 데이터로 페이지별 수익 기여도를 파악해 2단계 확장 우선순위를 정한다.
- 구체적인 트래픽·수익 목표치(월간 방문자 수, RPM, 목표 광고 수익 등)는 실측 데이터가 없는 상태라 이 문서에서 임의로 확정하지 않는다 — **1개월 차 실측 후 이 문서를 갱신해 목표 수치를 채워 넣는 절차를 권장한다.**

### 2단계 목표 (확장, 1단계 데이터 확보 후)

서비스 범위 확장은 아래 단계 중 어디까지 갈지 아직 미정이므로, 옵션으로 제시한다.

1. **범위 유지 확장**: 로또 6/45에 집중하되 도구 종류를 늘린다(예: 번호별 개인 통계, 특정 판매점 필터, 더 다양한 통계 뷰).
2. **복권 전반 확장**: 연금복권, 스피또 등 동행복권이 취급하는 다른 상품까지 다루는 "복권 도구 허브"로 넓힌다. 이 경우 F1~F4와 유사한 조사(공공데이터 유무, 통계 계산 방식)를 상품별로 새로 해야 한다.
3. 기존 PRD 2단계 기능(F10 구매 체크인, F11 당첨 확인 알림, F12 위치 배너)도 이 시점에 재검토한다 — 알림 기반 기능은 재방문율을 높여 광고 노출 총량을 늘리는 효과가 있어, 수익화 목표와도 직접 연결된다.

**최종 확정은 1단계 트래픽/피드백 데이터를 본 뒤 결정한다** — 이는 기존 PRD의 "1단계 출시 후 피드백을 모아 2단계 우선순위를 정한다"는 철학을 그대로 계승한 것이다.

### Non-goals (수익화 단계에서도 유지)

- 복권(로또) 발행·판매·구매 대행·양도 — 법적으로 명확히 금지된 영역이며, 광고 네트워크 정책(2절)에서도 별도 취급되는 민감 카테고리다. "번호 생성·통계·판매점 안내"라는 정보 제공자 역할을 절대 넘지 않는다.
- 당첨 확률을 높인다는 주장, 번호별 우열(별점·순위·추천) 표현 — 기존 F5 원칙 그대로 유지.
- 사행성을 조장하는 광고 소재(사설 도박, 불법 베팅 사이트 등)의 게재 — 3절 지침 참고.

## 3. 지침 (수익화 확장 시 반드시 지킬 원칙)

### 3.1 법적·정책 리스크 — 광고 네트워크별 확인 필요 (조사 결과 반영)

이번 조사에서 확인한 핵심 사실 두 가지가 있다.

- **카카오 광고(애드핏/톡채널검색광고 등)**: 카카오 비즈니스 광고 운영정책의 "제한 업종" 항목에 사행산업통합감독위원회법이 규정하는 '카지노업·경마·경륜·경정·**복권**·체육진흥투표권·소싸움 경기' 등이 명시적으로 포함돼 있고, "복권을 발행·판매하는 사이트", "복권 발행·판매 사이트로의 접속을 유도하는 배너·링크를 가진 사이트"는 광고 집행이 아예 불가능하다고 돼 있다. 이 규정은 검색광고 쪽 문서에서 확인한 것이라 디스플레이 광고(AdFit)에 그대로 동일하게 적용되는지는 별도 확인이 필요하지만, "복권"이 사행산업 규제 대상 업종으로 명시된다는 점 자체가 카카오 계열 광고를 1차 채널로 쓰기엔 리스크가 있다는 뜻이다. **카카오 광고를 검토한다면 신청 전에 반드시 최신 정책 원문과 실제 승인 사례를 재확인해야 한다.**
- **Google AdSense/Ads**: 2026년 3월과 9월(예정)에 '도박 및 게임' 카테고리 광고주 인증 요건이 확대되고 있지만, 이는 주로 **도박 상품을 광고하려는 광고주** 쪽 요건이고, 순수 정보/유틸리티 사이트가 애드센스 광고를 **게재(퍼블리셔)**하는 것을 막는 규정은 이번 조사에서 확인되지 않았다. 실제로 로또 번호 생성기류 웹사이트(`pyony.com`, `wepplication.github.io` 등)와 Google Play의 로또 번호 생성 앱들이 존재하며 이런 유형의 서비스가 시장에 통용되고 있다는 정황도 확인된다. 다만 이는 정황 증거이지 공식 승인 보장이 아니므로, **애드센스 신청 시 사이트가 복권 판매/도박이 아니라 통계·정보 도구임을 명확히 하고, 신청 후 반려되면 사유를 확인해 대응한다.**
- 결론: **1차 채널은 Google AdSense, 2차로 검토할 채널(제휴 링크, 다른 디스플레이 네트워크 등)은 최신 정책을 매번 개별 확인 후 진행한다.** 광고 네트워크 정책은 수시로 바뀌므로, 이 문서의 조사 내용을 "현재 시점 참고자료"로만 쓰고 실제 신청 직전에 각 네트워크 공식 문서를 다시 확인해야 한다.

### 3.2 광고 소재 자체에 대한 원칙

- 사이트에 자동으로 붙는 광고 중 **불법 사설 도박, 고위험 베팅 사이트 등 사행성 조장 광고는 차단**한다(Google AdSense의 "민감한 카테고리 차단" 등 콘텐츠 필터링 기능 활용). 로또 통계 도구라는 신뢰를 스스로 깨는 광고가 붙지 않도록 광고 카테고리를 능동적으로 관리한다.
- 광고 영역은 본문 UI(전략 카드, 결과 화면, 판매점 카드 등)와 시각적으로 명확히 구분하고 "광고"/"AD" 라벨을 표시한다 — 네이티브 광고가 F1의 "최종 조합 후보"나 F4의 "1등 배출 배지"처럼 보이지 않도록 한다. 이는 기존 F5의 "지어낸 근거로 우열을 암시하지 않는다" 원칙을 광고 영역에도 확장 적용하는 것이다.

### 3.3 제품·콘텐츠 지침

- "도구형 웹사이트"는 단일 페이지 앱이 아니라 **기능별로 독립된 URL(랜딩 페이지)**을 갖는 구조로 재구성한다 — 검색 유입이 수익의 전제 조건이므로, 각 기능(번호 생성기/예산별 생성/역대 통계/판매점 찾기)이 검색엔진에 개별적으로 색인될 수 있어야 한다.
- 각 랜딩 페이지는 위젯(생성 버튼, 지도)만 있는 게 아니라 검색 의도에 맞는 설명 텍스트·FAQ 등 콘텐츠 뎁스를 갖춘다 — 이는 SEO 품질 신호이자 애드센스 심사 통과 조건과도 맞닿아 있다.
- 텔레그램 채널은 폐기하지 않는다. 웹을 신규 유입(검색) 채널로, 텔레그램을 재방문·알림 채널로 역할을 나눈다.
- 기존 F5 법적 안전장치(최초 진입 안내, 상세 면책조항, 우열 표현 금지)는 신규 페이지에도 동일하게 적용한다 — 페이지가 늘어난다고 개별 페이지마다 문구를 빠뜨리지 않도록 공통 컴포넌트(`Disclaimer`, `DetailedDisclaimer`)를 재사용한다.

### 3.4 기술 지침

- 기존 `lib/lotto/*`(가중치·전략·통계 로직)와 `data/*`(정적 JSON 데이터 파이프라인) 자산은 그대로 재사용한다 — 이미 검증된 계산 로직을 다시 만들 필요는 없다.
- 신규 랜딩 페이지 확장 시에도 F4에서 세운 원칙(정적 배치 데이터 + 빌드 타임 캐시, 원본 데이터 갱신 주기에 맞춘 수동/예약 재적재)을 그대로 따른다.
- Next.js App Router의 메타데이터 API, `sitemap.xml`/`robots.txt`를 추가해 SEO 기반을 갖춘다.
- 광고 슬롯은 레이아웃 설계 초기 단계부터 반응형 공간을 예약해 삽입 시 레이아웃이 흔들리지 않도록 한다(Core Web Vitals 관리 — 이는 애드센스 승인·유지에도 영향을 준다).

### 3.5 로드맵 게이트 원칙

- 2단계(서비스 범위 확장, F10~F12 재검토)는 1단계 실측 데이터(트래픽, 광고 수익, 사용자 피드백) 없이 착수하지 않는다 — 기존 PRD가 F9(피드백 수집)를 1단계 필수 요건으로 못박은 것과 동일한 원칙이다.
- 광고 네트워크 정책은 수시로 바뀌므로, 매 분기 최소 1회 각 네트워크의 최신 정책을 재확인하는 절차를 둔다(기존 PRD가 판매점 데이터 갱신을 반기 점검으로 못박은 것과 같은 방식).
