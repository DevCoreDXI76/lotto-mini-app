# 로또랩(LottoLab) 1단계 실행 계획

> 이 문서는 `docs/PROJECT-CHARTER.md`(확장 기획서)를 브레인스토밍으로 구체화한 실행 계획이다. 기획서를 대체하지 않고, 기획서 4절 "다음 단계"를 실제로 순서화·체크리스트화한다. 아래 결정사항은 2026-07-23 브레인스토밍 세션에서 확정됐다.

## 1. 확정된 의사결정

| 항목 | 결정 |
|---|---|
| 브랜드명 | **로또랩 (LottoLab)** 확정. 옵션 B(신규 브랜드) 기각 — 리브랜딩을 정당화할 "2단계 확정"이 아직 없어 기획서 결론대로 옵션 A 채택. |
| 애드센스 신청 전략 | **완전한 콘텐츠 완성 후 신청이 아니라, 최소 골격으로 조기 신청 → 결과(승인/반려 사유)를 다음 단계 근거로 사용.** 승인 가정 위에 몇 주 작업을 먼저 태우는 리스크를 피한다. |
| 신규 랜딩페이지 우선순위 | **회차별 개별 페이지(프로그래매틱 SEO)를 1단계에 포함**(F3 역대 통계의 심화라 게이트 대상 아님). 지역별 판매점 페이지 · 세금 계산기 · 내 번호 확인은 **2단계 백로그**로 보류. |
| 2차 수익화 채널 | 지금은 후보만 유지(제휴 링크, 네이버 애드포스트 등), 실행은 1단계 실측 데이터 확보 후 3.5절 게이트 통과 뒤 결정. |

### 짚고 넘어갈 충돌 지점 — 도메인 문제

브랜드명은 로또랩으로 확정했지만, **도메인을 새로 사서 옮기는 것은 "조기 신청" 전략과 충돌한다.** 새 도메인은 신뢰도(사이트 나이)를 처음부터 다시 쌓아야 해서, 지금 갖고 있는 `lotto-mini-app-three.vercel.app`의 기존 이력(있다면)을 버리는 셈이다. 이번 Phase 1에서는 **도메인은 유지하고 "로또랩"은 표시 브랜드명(타이틀·헤더·문구)으로만 우선 적용**하는 것을 제안한다. 커스텀 도메인 구매는 실측 데이터를 보고 별도로 결정할 것(연결된 열린 질문, 4절 참고).

## 2. 리서치로 확인된 리스크 — 계획에 반영

- **Google의 "scaled content abuse"(대량 생산 콘텐츠 남용) 정책**: 숫자만 다른 동일 템플릿 페이지를 대량 생성하면 스팸으로 분류될 수 있다는 것이 Google 공식 정책이다. 회차별 페이지를 만들 때 "회차 번호 + 당첨번호"만 나열하는 게 아니라, **회차마다 실제로 다른 정보**(그 회차의 홀짝비율, 번호 합계, 직전 회차 대비 변화 등)를 자동 계산해 넣어야 한다. 이 값들은 이미 `lib/lotto/analysis.ts`, `weights.ts`로 계산 가능해서 새 로직이 필요 없다.
- **(비공식 가이드 기준, Google 공식 확인 아님 — 참고치)**: 승인 전 최소 20개 이상의 고유 콘텐츠, 사이트 운영 3개월 이상이 유리하다는 통설이 있다. 공식 수치가 아니므로 **신청 직전 애드센스 공식 문서를 다시 확인**해야 한다(기존 지침 3.1 "매번 개별 확인" 원칙과 동일).
- **YMYL(Your Money or Your Life) 성격**: 로또/복권 주제는 금전과 밀접해 Google이 전문성·신뢰성(E-E-A-T) 신호를 더 엄격히 볼 수 있는 카테고리다. 기존 F5 `Disclaimer`/`DetailedDisclaimer` 컴포넌트가 이미 "과장하지 않고 투명하게 밝힌다"는 역할을 하고 있는데, 이게 심사에도 긍정적으로 작용할 가능성이 있다 — 별도 대응 불필요, 유지만 하면 됨.

## 3. 로드맵

### Phase 1a — 최소 골격 준비 (지금 ~ 1~2주)

- [ ] 브랜드명 로또랩 적용: `app/layout.tsx` 전역 메타데이터, 각 페이지 타이틀, README, (필요시) 텔레그램 채널 소개문구
- [ ] 페이지별 개별 메타데이터(Next.js Metadata API) — 지금 전역 하나뿐인 title/description을 `/`, `/generator`, `/stats`, `/stores` 각각에 맞게 분리
- [ ] `sitemap.xml`, `robots.txt` 추가
- [ ] 각 페이지에 검색 의도에 맞는 설명 텍스트(1~2문단) + FAQ 2~3개 추가 — 지침 3.3 요건이자 위 scaled content 리스크 완화책
- [ ] 광고 슬롯 자리만 반응형으로 예약(실제 광고 코드는 승인 후 삽입, Core Web Vitals 흔들림 방지)
- [ ] Search Console 소유권 확인, robots 접근성, 모바일 반응형 기본 점검
- [ ] 위 항목 완료 시점에 애드센스 공식 최신 요건 페이지 재확인(통설이 아닌 공식 기준으로 최종 점검)
- [ ] → **신청**

### Phase 1b — 결과 대응

- **승인**: 광고 슬롯에 실제 코드 삽입 → "민감한 카테고리 차단" 필터링 설정(불법 사설 도박/고위험 베팅 차단, 지침 3.2) → 콘텐츠 계속 확장하며 1개월 실측 데이터 수집 시작
- **반려**: 반려 사유 확인 → 그 사유를 체크리스트로 변환해 재신청 준비 (예: "콘텐츠 부족"이 사유면 아래 회차별 페이지 작업을 앞당겨 보강)

### Phase 1 병행 — 회차별 개별 페이지 (프로그래매틱 SEO)

F3(역대 통계)의 심화 기능이라 신규 서비스 확장이 아님 → 게이트 대상 아님, Phase 1a와 병행 착수 가능.

- [ ] `/result/[회차번호]` 라우트 설계, `data/lotto-full-history.json` 기반 `generateStaticParams`로 정적 생성
- [ ] 페이지별 고유 정보 자동 계산해 삽입(홀짝비율·총합·직전 회차 대비 등) — `lib/lotto/analysis.ts`, `weights.ts` 재사용
- [ ] 회차 목록 인덱스 페이지(페이지네이션) + `/stats`와 상호 내부링크
- [ ] F5 면책조항 컴포넌트(`Disclaimer`) 동일 적용

### Phase 2 백로그 (게이트: 1단계 실측 트래픽 · 광고수익 · 피드백 확보 후 재우선순위 확정)

1. 지역별 판매점 페이지(`/stores/[지역]`)
2. 로또 당첨금 세금 계산기
3. 내 번호 확인(역방향 조회 도구, `lib/lotto/analysis.ts` 재사용 가능)
4. 2차 수익화 채널(제휴 링크, 네이버 애드포스트 등) 실행 여부 결정

## 4. 열린 질문

- 커스텀 도메인(로또랩 관련 `.com`/`.co.kr`)으로 옮길지, 기존 `lotto-mini-app-three.vercel.app`을 유지할지 — 위 "도메인 문제" 참고. 조기 신청 전략과의 정합성 때문에 **유지를 우선 추천**하나 최종 확정은 필요.
- 애드센스 공식 최소 요건 최신본은 신청 직전 다시 확인(비공식 통설 수치는 이 문서에 확정치로 반영하지 않음).
