# Phase 1 작업 인계 — 트레이드오프 및 다음 단계

> Cowork 세션(2026-07-23)에서 진행한 로또랩(LottoLab) 확장 작업을 Claude Code(로컬 CLI)로 이어가기 위한 인계 문서다. 관련 문서: `docs/PROJECT-CHARTER.md`(확장 기획서), `docs/PHASE1-EXECUTION-PLAN.md`(실행 계획, 이 문서보다 먼저 작성됨).

## 0. 지금 상태 — 검증 안 된 변경사항이 이미 파일시스템에 적용돼 있음

Cowork 세션에서 아래 10개 파일을 이미 로컬에 반영했는데, `tsc`/`lint`/`test`/`build` 중 아무것도 돌리지 못한 채로 기기 연결이 끊겨 중단됐다. **Claude Code에서 제일 먼저 할 일은 이 검증이다** — 이미 커밋된 변경이 아니라 워킹 트리에 떠 있는 상태이니, `git status`/`git diff`로 먼저 실제 변경 범위를 확인할 것.

| 파일 | 상태 | 내용 |
|---|---|---|
| `lib/site.ts` | 신규 | `SITE_NAME`, `SITE_URL` 상수 |
| `app/layout.tsx` | 수정 | 브랜드명 로또랩 적용, `metadataBase`/title template 추가, `lang="en"`→`"ko"` |
| `app/page.tsx` | 수정 | h1 "로또 미니앱"→"로또랩", 서브타이틀 한 줄 추가 |
| `app/generator/page.tsx` | 수정 | 클라이언트 로직 제거하고 서버 컴포넌트(메타데이터 전담)로 교체 |
| `app/generator/GeneratorClient.tsx` | 신규 | 기존 `app/generator/page.tsx`의 클라이언트 로직 그대로 이동 (`'use client'`) |
| `app/stores/page.tsx` | 수정 | 위와 동일한 패턴으로 서버 컴포넌트로 교체 |
| `app/stores/StoresClient.tsx` | 신규 | 기존 `app/stores/page.tsx`의 클라이언트 로직 이동 |
| `app/stats/page.tsx` | 수정 | 이미 서버 컴포넌트였어서 `export const metadata`만 추가 |
| `app/sitemap.ts` | 신규 | Next.js `MetadataRoute.Sitemap` — `/`, `/generator`, `/stats`, `/stores` |
| `app/robots.ts` | 신규 | Next.js `MetadataRoute.Robots` |

검증 체크리스트:

- [ ] `git status` / `git diff`로 변경 범위 확인
- [ ] `npx tsc --noEmit`
- [ ] `npm run lint`
- [ ] `npm run test` (vitest — 특히 `GeneratorClient`/`StoresClient`로 이름을 바꾼 컴포넌트를 참조하는 테스트가 있었는지 확인)
- [ ] `npm run build` (Next.js 빌드 타임에 `generateStaticParams` 등 이슈 없는지, sitemap/robots 라우트가 정상 생성되는지)
- [ ] 문제없으면 의미 단위로 git commit

## 1. 오늘 확정된 의사결정과 트레이드오프

### 1.1 애드센스 신청 순서 — 완전한 콘텐츠 완성 후 vs 최소 골격 조기 신청

**결정: 최소 골격으로 조기 신청.** 승인 가정이 맞다는 전제로 몇 주치 콘텐츠 작업을 먼저 태우는 것보다, 지금 있는 최소 구조로 신청해서 반려 사유(있다면)를 실제 체크리스트로 받는 게 더 빠르다고 판단했다. 트레이드오프: 승인 가능성 자체는 이 선택으로 달라지지 않는다 — 단지 "학습 시점"을 앞당기는 베팅이다. 반려되면 재신청까지 시간이 걸릴 수 있다는 점(정책마다 재신청 대기 기간이 있을 수 있음, 확인 필요)이 이 베팅의 비용이다.

### 1.2 브랜드명 — 로또랩(LottoLab) vs 로또허브(LottoHub) vs 신규 브랜드(드로우랩 등)

**결정: 로또랩.** 신규 브랜드는 리브랜딩 비용을 정당화할 "2단계 확정"이 아직 없어 기각. 로또허브 대비 로또랩을 택한 이유는 "여러 서비스 모음"보다 "담백하게 분석하는 도구"라는 톤이 F5(우열 표현 금지) 원칙과 더 잘 붙는다는 판단 — 이건 취향 판단이라 번복 여지가 있다.

### 1.3 도메인 — 기존 유지 vs 로또랩 관련 신규 도메인 구매

**결정 (제안 단계, 최종 확정 아님): 기존 `lotto-mini-app-three.vercel.app` 유지.** 새 도메인은 사이트 나이(신뢰도)가 리셋돼 1.1의 "조기 신청" 전략과 정면으로 충돌한다. `lib/site.ts`의 `SITE_URL`이 기존 도메인으로 하드코딩돼 있는데, 이후 도메인을 바꾸기로 하면 이 파일 하나만 고치면 되도록 의도적으로 분리해뒀다.

### 1.4 신규 랜딩페이지 우선순위 — 회차별 개별 페이지 우선, 나머지 3개(지역별 판매점/세금계산기/내번호확인)는 2단계 보류

**결정: 회차별 페이지(`/result/[회차번호]`)만 1단계에 포함.** 기존 `data/lotto-full-history.json`을 그대로 라우트로 펼치는 프로그래매틱 SEO라 새 데이터·새 계산 로직이 필요 없고, F3(역대 통계)의 심화라 3.5절 게이트("2단계는 1단계 실측 데이터 없이 착수 안 함") 대상이 아니라고 판단했다. 나머지 3개(지역별 판매점 페이지, 세금 계산기, 내 번호 확인)는 사용자가 명시적으로 "2단계"로 분류했다 — 아직 구현 안 됨, 게이트 통과 후 우선순위 재확정 필요.

**중요 — 아직 코드로 반영 안 됨**: 회차별 페이지 자체는 이번 세션에서 구현하지 않았다(0절 파일 목록에 없음). 착수 시 아래 리스크를 반드시 반영할 것.

### 1.5 `/generator`, `/stores`의 메타데이터 처리 — 서버/클라이언트 분리 vs 다른 대안

두 페이지 다 원래 전체가 `'use client'`였는데, Next.js는 `'use client'` 파일에서 `metadata`를 export할 수 없다. 검토한 대안과 선택 이유:

- **채택: `page.tsx`(서버, 메타데이터) + `XxxClient.tsx`(클라이언트, 기존 로직) 분리.** 가장 표준적인 패턴이고 기존 클라이언트 로직을 한 줄도 안 건드리고 그대로 옮길 수 있어 리스크가 가장 적었다.
- 기각: 페이지를 서버 컴포넌트로 완전히 재작성하고 상태가 필요한 부분만 작은 클라이언트 서브컴포넌트로 쪼개는 방식 — 더 "깔끔"할 수 있지만 오늘 범위(메타데이터 추가)에 비해 리팩터링 범위가 커서 보류.

### 1.6 텔레그램 브랜드명 통일 여부 — 미해결, 다음 세션에서 사용자 확인 필요

`lib/telegram/api.ts`에는 여전히 "로또 미니앱"이 남아 있다. 의도적으로 손대지 않았다 — 텔레그램은 이미 그 이름으로 구독자가 있는 채널이라, 웹만 로또랩으로 전환하고 텔레그램은 기존 이름을 유지하는 게 맞다고 잠정 판단했다. **사용자에게 최종 확인을 아직 못 받은 상태**이니, Claude Code 세션에서 이 부분을 진행하기 전에 다시 물어볼 것.

### 1.7 2차 수익화 채널 — 지금 결정 안 함

3.5절 로드맵 게이트 원칙에 따라, 제휴 링크·네이버 애드포스트 등은 후보만 나열하고 실행은 1단계 실측 데이터 확보 후로 미뤘다. 지금 시점에 코드/설정 변경 없음.

## 2. 리서치로 확인된 리스크 (신청 직전 재확인 필요)

- **Google Search "scaled content abuse" 정책**: 회차별 페이지를 단순 데이터 나열(회차 번호 + 당첨번호만)로 만들면 스팸으로 분류될 위험이 있다는 게 Google 공식 정책이다. 회차별 페이지를 실제로 구현할 때 각 페이지에 고유 정보(홀짝비율, 총합, 직전 회차 대비 등 — `lib/lotto/analysis.ts`/`weights.ts`로 계산 가능)를 반드시 자동 삽입할 것.
- **(비공식 가이드, 공식 확인 아님)**: 승인 전 최소 20개 이상 고유 콘텐츠, 사이트 운영 3개월 이상이 유리하다는 통설. 애드센스 공식 문서 기준으로 신청 직전 재확인 필요.
- 출처: [Google Search 공식 스팸 정책 — Scaled Content Abuse](https://developers.google.com/search/docs/essentials/spam-policies), [애드센스 승인 방법 2026년 심사 기준 (비공식)](https://jeff-info.co.kr/sidehustle/adsense-approval-guide-2026/)
